# FinOS ‚Äî Budget Engine

Monthly / weekly / custom-period budgets with category and global caps. Utilization,
overspend, remaining, and projected exhaustion are computed on read from the ledger;
threshold alerts are generated by consuming `TransactionCreated` events.

Code: pure [`app/domain/budgets.py`](../backend/app/domain/budgets.py); persistence
[`app/modules/budgets/`](../backend/app/modules/budgets).

---

## Entities

| Entity | Table | Notes |
|---|---|---|
| Budget | `budgets` | name, period_type, currency, `overall_amount_minor` (global cap), `custom_period_days` |
| BudgetCategoryAllocation | `budget_allocations` | per-category cap |
| BudgetAlert | `budget_alerts` | system-generated, append-only (warning / over) |
| BudgetPeriod | *(computed)* | period bounds derived from `period_type` + `as_of` |

## Period model (deterministic)

`period_bounds(budget, as_of, offset)` returns the `[start, end]` dates for the period
`offset` from the current one (offset `0` = current, `-1` = previous, ‚Ä¶):

- **monthly** ‚Üí first‚Ä¶last of month.
- **weekly** ‚Üí Monday‚Ä¶Sunday.
- **custom** ‚Üí fixed `custom_period_days` windows anchored at `created_at`.

Historical performance is just `get_status(..., offset=-n)` ‚Äî no stored history table.

## Utilization model

`assess_line(allocated, spent, period_start, period_end, as_of)`:

- **remaining** = allocated ‚àí spent (negative = overspent).
- **utilization_ratio** = spent / allocated (None when allocation is 0).
- **health** ‚àà `{under, warning, over}` (warning at ‚â•80% by default).
- **projected_spend** = run-rate `spent √ó total_days √∑ elapsed_days`.
- **projected_exhaustion** = the date the run-rate reaches the allocation (None if it won't).

`assess_budget` rolls category lines up into a total. The brief's example (‚Çπ5000 food,
‚Çπ4200 spent ‚Üí ‚Çπ800 remaining, 84%) is exactly `assess_line`, verified in tests.

Spend is sourced from the ledger via the reporting engine (`spending_by_category` over the
period), so a budget always reflects real transactions.

## Event integration (alerts)

The outbox dispatcher routes `TransactionCreated` to `on_transaction_created`, which calls
`evaluate_alerts(user_id, category_id)`. That recomputes the affected category's status and
inserts a `BudgetAlert` when a threshold is first crossed. Alerts are **idempotent** ‚Äî
unique per `(budget, period, category, level)` ‚Äî so at-least-once delivery is safe.

## API

`POST/GET/PATCH/DELETE /v1/budgets` ¬∑ `GET /v1/budgets/{id}/status?period_offset=` ¬∑
`GET /v1/budgets/alerts`. Filtering, pagination, validation, OpenAPI included.

## Design rationale, tradeoffs, ADRs

- **Computed status (ADR-011):** utilization is derived, never cached, so it cannot drift
  from the ledger. Tradeoff: a per-period aggregate query on read (indexed on
  `(user_id, category_id, occurred_at)`).
- **Alerts are persisted, status is not:** an alert is a *fact that happened at a moment*
  (needs a durable record + notification); utilization is a *current view* (derive it).
  This split keeps the model honest.
- **Category-vs-overall rollup:** if a budget has category allocations, the rollup uses them
  and shows any `overall` cap as an informational line, avoiding double counting.

## Future extension points

- Rollover budgets (carry unspent allocation to the next period).
- Per-budget warning ratios (the engine already parameterizes it).
- Push notifications on `BudgetAlert` creation via a notifications module.
